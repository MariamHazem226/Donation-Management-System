Responsibilities:

- Adds JwtAuthenticationFilter into the filter chain.
Important Methods:
- filterChain(HttpSecurity http): creates SecurityFilterChain.
Connected With:
- security/JwtAuthenticationFilter.java
- security/CustomOAuth2UserService.java
- security/OAuth2LoginSuccessHandler.java


----------------------------------------
FILE: backend/src/main/java/com/givinghands/givinghands/security/JwtService.java
----------------------------------------
Purpose:
Generates and validates JWT tokens.
Responsibilities:
- Create JWT using user email as subject.
- Store role and additional claims (userId, role, name).
- Validate expiration.
Important Methods:
- generateToken(User user)
- parseClaims(String token)
- isValid(String token)
- getEmailFromToken(String token)
Connected With:
- controller/AuthController (token generation)
- security/JwtAuthenticationFilter (token validation)


----------------------------------------
FILE: backend/src/main/java/com/givinghands/givinghands/security/JwtAuthenticationFilter.java
----------------------------------------
Purpose:
Reads Authorization: Bearer <token> and sets the SecurityContext.
Responsibilities:
- Extract JWT from request headers.
- Validate JWT and load the user using UserRepository.
- Create Spring Security Authentication with ROLE_<ROLE> authority.
- Put it into SecurityContextHolder.
Important Methods:
- doFilterInternal(...): core request filter.
Connected With:
- security/JwtService
- repository/UserRepository
- SecurityConfig (registers this filter)

Connected With:
- service/AuthService

- security/OAuth2RoleHolder

BACKEND: ARCHITECTURE EXPLANATION (BEGINNER-FRIENDLY)
========================================

----------------------------------------
1) How Authentication Works
----------------------------------------
There are two authentication paths:

A) Email/Password
1. Frontend calls POST /api/auth/register
   - AuthController.register() calls AuthServiceImpl.registerUser(...).
   - User is saved with a BCrypt-hashed password.
   - JWT is generated in AuthController using JwtService.generateToken(user).
2. Frontend calls POST /api/auth/login
   - AuthController.login() calls AuthServiceImpl.loginUser(...).
   - Password is checked with BCrypt.
   - JWT is generated and returned.

B) Google OAuth
1. Frontend navigates to /api/auth/oauth/google?role=...
2. AuthController.googleOAuth(...)
   - Stores requested role in session using OAuth2RoleHolder.
   - Redirects the browser to Spring Security OAuth2 start URL: /oauth2/authorization/google.
3. OAuth2LoginSuccessHandler and CustomOAuth2UserService handle account creation/update and login.
4. A JWT is expected to be issued after successful OAuth flow (details may be in OAuth2LoginSuccessHandler).

C) JWT Request Authentication
1. For every request, JwtAuthenticationFilter checks Authorization header.
2. If token is valid:
   - It loads the user by email from the token.
   - It sets SecurityContextHolder authentication with ROLE_<role> authority.

Key files involved:
- SecurityConfig
- JwtAuthenticationFilter
- JwtService
- AuthController
- AuthServiceImpl


----------------------------------------
2) Campaign Flow (Create → Admin Approve → Notifications)
----------------------------------------
A) Creating a campaign
1. POST /api/campaigns with CampaignDTO.
2. CampaignController delegates to CampaignServiceImpl.createCampaign().
3. CampaignServiceImpl validates:
   - title required
   - goal > 0
   - deadline required
   - organizationId required
4. It also restricts creation so only ORGANIZATION or ADMIN roles can create.
5. Campaign is saved with status = PENDING.
6. After saving, it publishes NotificationEvents.ADMIN_NEW_CAMPAIGN_REQUEST.

B) Admin approves or rejects
1. Admin calls PUT /api/admin/campaigns/{id}/approve (AdminController.approveCampaign).
2. AdminServiceImpl.approveCampaign():
   - sets status APPROVED
   - saves campaign
   - calls publishApprovalNotifications(saved)
3. publishApprovalNotifications publishes:
   - ORG_CAMPAIGN_APPROVED
   - NEWSLETTER_CAMPAIGN_APPROVED_ACTIVE
   - and if campaign allows volunteers: NEWSLETTER_VOLUNTEER_OPPORTUNITY
4. If admin rejects:
   - PUT /api/admin/campaigns/{id}/reject (AdminController.rejectCampaign)
   - AdminServiceImpl.rejectCampaign() sets status REJECTED
   - publishes ORG_CAMPAIGN_REJECTED

Key files involved:
- CampaignController
- CampaignServiceImpl
- AdminController
- AdminServiceImpl
- EmailNotificationListener
- notification events/constants


----------------------------------------
3) How Donations Work (Donation → Funding update → Goal reached notifications)
----------------------------------------
1. Frontend calls POST /api/donations with DonationDTO.
2. DonationController calls DonationServiceImpl.submitDonation().
3. DonationServiceImpl:
   - saves Donation record (status COMPLETED)
   - publishes USER_DONATION_SUBMITTED
   - updates Campaign currentAmount
4. If updated currentAmount >= goalAmount:
   - publishes NEWSLETTER_CAMPAIGN_GOAL_REACHED
   - publishes ADMIN_CAMPAIGN_GOAL_REACHED
5. EmailNotificationListener receives these events and triggers notification service methods.

Key files involved:
- DonationController
- DonationServiceImpl
- NotificationEvents
- EmailNotificationListener


----------------------------------------
4) How Admin Approval Works
----------------------------------------
Admin logic is centralized:
- AdminController: HTTP endpoints
- AdminServiceImpl: business logic

When admin approves:
- Campaign status changes in DB.
- Notifications are published through NotificationPublisher/Observer hub.

This keeps business logic decoupled from email delivery.


----------------------------------------
5) How the Notification System Works (Observer pattern + email services)
----------------------------------------
The notification system is built around a simple Observer pipeline:

Step A: Publish
- Domain services (AuthServiceImpl, CampaignServiceImpl, DonationServiceImpl, VolunteerServiceImpl, AdminServiceImpl)
  call notificationPublisher.publish(eventType, payload).

Step B: Observer hub
- NotificationPublisher forwards to NotificationEventManager.notify(eventType, data).
- NotificationObserverConfig registers listeners, including EmailNotificationListener.

Step C: Listener dispatch
- EmailNotificationListener.update(eventType, data):
  - Switches by event type string constants
  - Extracts IDs from payload
  - Calls the correct notification service method:
    - NewsletterNotificationService
    - OrganizationNotificationService
    - UserNotificationService
    - AdminNotificationService

Step D: Email sending
- The specific notification service implementations will use EmailService and Thymeleaf templates in
  backend/src/main/resources/templates/emails/...

Important files:
- NotificationPublisher
- NotificationObserverConfig
- EmailNotificationListener
- NotificationEvents
- notification handler (NotificationDispatcher)


----------------------------------------
6) How Observer Pattern is Structured
----------------------------------------
There are two observer-related layers in the project:

A) Generic CampaignEventManager / CampaignEventListener
- pattern/observer/CampaignEventManager.java
- pattern/observer/CampaignEventListener.java
- pattern/observer/CampaignPublisher.java

B) Application notification observer hub (used at runtime)
- notification/NotificationPublisher.java
- pattern/observer/NotificationEventManager.java (not opened here but referenced)
- pattern/observer/NotificationListener.java
- pattern/observer/EmailNotificationListener.java
- pattern/observer/LoggingNotificationListener.java

In practice for notification emails, the project uses NotificationEventManager + listeners.


----------------------------------------
7) How JWT Authentication Works (Summary)
----------------------------------------
- JwtService signs tokens with HMAC using app.jwt.secret.
- JwtAuthenticationFilter validates signature + expiration.
- It loads User by token subject (email) and sets role authority.


----------------------------------------
8) How Frontend Communicates with Backend
----------------------------------------
The frontend pages are static HTML served from backend/src/main/resources/static/
(and also mirrored in the top-level frontend/ folder).

Communication model:
- The frontend uses JavaScript (frontend/js/main.js) to call backend REST APIs.
- Typical endpoints used:
  - /api/auth/register
  - /api/auth/login
  - /api/auth/me
  - /api/campaigns (list/create/update/delete)
  - /api/donations (submit + history)
  - /api/admin/* (dashboard + moderation)
  - /api/notifications/* (currently placeholder)

Authorization model:
- The frontend expects a JWT token returned from AuthController.
- It then sends it in request header Authorization: Bearer <token>.
- JwtAuthenticationFilter handles it.
